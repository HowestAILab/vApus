Features
  • The power of vApus

    When clients come to us, they generally have two questions:
    How will our specific application scale?
    How will our hardware perform under increasing load?

    We can actually offer an answer to those questions. Not by setting up a complicated standardized infrastructure like that of TPC (Transaction Processing Performance Council), but by using our unique software: vApus.

    Setting up a test requires only one PC or notebook. What vApus does is simulating a configurable number of users and letting them test with real data from an imported log of the application.


    This way, it is easy to see how the application performs, and bottlenecks can be easily detected. Because it is possible to set the number of user to test with, future scaling can also be determined.

    How the hardware reacts and performs under a specific load can be monitored using vApus in combination with our server monitoring toolset.
    These tools will log their counters to vApus, which can link them to the number of users being tested with. These counters can be anything, ranging from CPU or memory usage to power usage.
  • What can be tested?

    Databases:
    When something is wrong with the communication flow between the application and the database server, that will mostly result in a performance issue. Since vApus is able to test all common databases (MySql, SqlServer, Oracle, ...) and communication with these servers can easily be logged and imported into vApus, bottlenecks can easily be determined.

    Webservers:
    Webdriven applications (for example: a portal site), can behave strangely if too many users are connecting to it. Before launching the application, it is vital to be sure whether it will withstand heavy stress.

    Socketservers:
    Definition: Every server that communicates over a socket, even when not using a standard defined protocol like HTTP or OLEDB. Contact Sizing Servers for more information about the creation of custom connection proxies for your socket-based communication.

    Using a sophisticated master/slave mechanism, virtual servers can be tested as well, putting every virtual machine under stress at the same time.

    The impact the different virtual machines have on each other, and the difference between native and virtual servers on the same hardware can easily be determined.

    The basic principle behind this method is actually using multiple instances of vApus (slaves), and letting them run on different clients, making sure they will not interrupt each other. These slaves are managed by a master instance of vApus.

    During the test, results are collected from the slaves and interpreted by the master.
